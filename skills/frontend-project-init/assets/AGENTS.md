## 编码约定

- 优先沿用现有代码风格，不做无关格式化。
- 新增代码应贴近所在文件的既有风格。
- 如果项目对某类文件有专用 skill 或本地说明，编辑或新建该类文件前先使用它。
- 优先使用项目内已引入的第三方库和现有工具能力，不要重复造轮子。
- 处理日期、金额、附件、枚举映射、下载/预览、请求封装等通用逻辑时，先检索项目已有工具、组件、依赖或内部能力；确认没有合适能力后，才允许新增小范围实现，并在最终说明中说明原因。
- 已有内部能力优先复用，不要重复实现已有基础组件或通用工具。
- 不要过度拆分函数或声明一次性函数；没有复用价值的短逻辑尽量保持内聚，放在最贴近使用处。
- 修改路由、鉴权、请求、全局配置时需要特别谨慎，因为影响面较大。
- 不要盲目做字段兼容；字段来源、含义、兼容范围或数据形态不清楚时，先问用户，不要猜。

### 注释与文风约定（AI 输出）

- 在不改变行为的前提下，优先按“功能分区”组织代码：类型/常量配置/状态/纯工具/数据请求/交互方法/展示格式化。
- 单次新增的同功能代码尽量放在同一连续区域或少数相邻区域，避免把新增逻辑分散到文件各处。
- 分区注释使用简短中文短句，风格统一为“xx 区：用途说明”。
- 新增功能代码的第一行应添加功能注释，说明该区域的业务用途或复杂约束。

  示例：

  ```js
  /* ================= 状态区：维护页面筛选和选中项 ================= */
  const filterForm = reactive({
    keyword: '',
    status: '',
  })
  ```

- 注释只解释结构意图、业务用途和复杂约束，不写显而易见的逐行注释。
- 重构以“可读性提升 + 低风险”为优先，避免为注释分区引入行为变化。

## 协作原则

- 开始修改前先查看相关文件和调用链，避免凭猜测改动。
- 新增功能时，相关代码尽量集中放在同一预期范围内，不要把单次新增逻辑分散到多个无关位置。
- 保留用户或本地已有未提交改动，不要执行 `git reset --hard`、`git checkout --` 等破坏性命令。
- 如果工作区已有与任务无关的改动，只关注本次任务涉及的文件。
- 涉及登录态、环境地址、代理、构建产物的改动，应在最终说明中明确风险和验证方式。
- 完成代码修改后，优先选择更小范围的验证命令或静态检查；不要每次都运行完整构建。仅在用户明确要求、改动影响构建配置/全局入口/依赖关系，或改动风险确实需要生产构建验证时运行构建。
- 不要主动运行项目说明中标记为不应主动运行的检查命令，除非用户明确要求。

## 提交前检查

建议在提交前确认：

- `git status --short` 中只有预期文件变更。
- 不应提交的本地配置、登录态、token、Cookie、个人环境配置没有进入提交。
- 通用逻辑已优先复用项目现有工具、组件、依赖或内部能力，没有无必要的手写实现。
- 关键页面、路由或行为已按需验证。
- 更小范围验证命令已按需执行，并记录结果；完整构建和类型检查仅在用户明确要求或确有必要时执行。

## 提交信息规范

- 每次提交标题都使用 `type: 中文标题` 格式，简短说明本次改动的核心目的。
- 提交类型优先使用 `feat`、`fix`、`refactor`、`docs`、`style`、`test`、`chore`，按实际改动选择。
- 每次提交都添加具体描述，使用 2-4 条中文列表说明主要改动、行为影响和必要的兼容处理。
- 提交信息不要包含本地 token、Cookie、环境地址、个人调试值等敏感内容。

## Karpathy 编码指南

> 来源：[andrej-karpathy-skills](https://github.com/multica-ai/andrej-karpathy-skills)
> 基于 Andrej Karpathy 对 LLM 编码陷阱的观察总结。这些指南偏向谨慎而非速度，简单任务自行判断。

### 1. 编码前思考

**不要假设。不要隐藏困惑。呈现权衡。**

- 明确说明假设——不确定时询问，不要猜测
- 存在多种解释时，呈现出来——不要默默选择一种
- 存在更简单的方法时，说出来——必要时提出异议
- 遇到不清楚的地方，停下来，指出困惑并要求澄清

### 2. 简洁优先

**用最少的代码解决问题。不要过度推测。**

- 不要添加要求之外的功能
- 不要为一次性代码创建抽象
- 不要添加未要求的"灵活性"或"可配置性"
- 不要为不可能发生的场景做错误处理
- 如果 200 行能写成 50 行，重写它

**检验标准：** 资深工程师会觉得过于复杂吗？如果是，简化。

### 3. 精准修改

**只碰必须碰的。只清理自己造成的混乱。**

- 不要"改进"相邻的代码、注释或格式
- 不要重构没坏的东西
- 匹配现有风格，即使你更倾向于不同的写法
- 注意到无关的死代码，提一下——不要删除它
- 删除因你的改动而变得无用的导入/变量/函数
- 不要删除预先存在的死代码，除非被要求

**检验标准：** 每一行修改都应该能直接追溯到用户的请求。

### 4. 目标驱动执行

**定义成功标准。循环验证直到达成。**

| 不要这样做 | 转化为 |
|---|---|
| "添加验证" | "为无效输入编写测试，然后让它们通过" |
| "修复 bug" | "编写重现 bug 的测试，然后让它通过" |
| "重构 X" | "确保重构前后测试都能通过" |

多步骤任务说明计划：

```text
1. [步骤] -> 验证: [检查]
2. [步骤] -> 验证: [检查]
3. [步骤] -> 验证: [检查]
```
